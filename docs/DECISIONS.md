# Decisions

Record here any choice not already covered by `TECHNICAL.md` or `ClaudeCodeInstruction.md`, in the order made.

## Phase 1 — packages/core

- **Merkle leaves.** The tree is built directly over the per-file SHA-256 hashes (sorted by path), not over a re-hash of `path\nhash\n`. TECHNICAL.md §3 says "binary Merkle tree over the sorted leaf hashes," which reads most naturally as the file hashes themselves; `project_sha256` already covers path+hash binding.
- **Merkle proof shape.** `merkleProof` returns `{ hash, position: "left" | "right" }[]`, sibling hashes ordered leaf→root, `position` meaning which side the sibling sits on. Not specified in TECHNICAL.md; chosen as the simplest shape that lets `verifyMerkleProof` recompute the root without extra lookups.
- **`sha256Hex`/Merkle/`projectHash` are async.** They use `crypto.subtle.digest`, which is inherently Promise-based in the Web Crypto API: forcing sync would mean a non-Web-Crypto fallback, which the phase brief rules out ("use the Web Crypto API").
- **`manifest.created` validation.** Required to be RFC 3339 / ISO-8601 with an explicit offset (zod `.datetime({ offset: true })`), matching the `Z`-suffixed example in TECHNICAL.md §3.
- **`estimateCellCost`.** Computes `full`/`compact` from the *actual* `encodeManifest` byte length (+61 CKB overhead) rather than the approximate `90 + 110/file` formula in TECHNICAL.md §3 — that formula is explicitly a ballpark ("≈"), and the real encoded size is available and exact.
- **`network.ts` invalid env value.** An unrecognized `VERICELL_NETWORK`/`VITE_VERICELL_NETWORK` value falls back to the `testnet` default rather than throwing, so a typo in deployment config fails safe (never silently defaults to `mainnet`).

## Phase 2 — packages/chain

- **v1 (no Type ID) cells remain readable forever.** `fetchProof`/`findVeriCells` never assume a type script: they read the manifest straight out of `outputsData` and check liveness by out point. A pre-Type-ID v1 cell and a post-Type-ID cell are both just "a live/dead cell with VeriCell JSON in its data" to every reader in this package — no migration or dual-path branching needed.
- **`buildAnchorTx`/`buildAnchorTxWithTypeId`/`buildWithdrawTx` take a `client` + `lock`, not a `Signer`.** Capacity and fee completion run against a `ccc.SignerCkbScriptReadonly(client, lock)` — a real signing key is never required to build the transaction. This lets the exact same builder serve a locally-connected wallet signer (web/CLI) and a server that only knows the payer's lock (Phase 5's `/proofs/prepare`, which must build an unsigned tx for a client to sign elsewhere).
- **Type ID first-version flow.** Per TECHNICAL.md §5, args are computed via `ccc.hashTypeId(tx.inputs[0], outputIndex)`. Since args depend on having an input, `completeInputsAtLeastOne` runs *before* the output (and therefore the type script) is added — mirroring `@ckb-ccc/type-id`'s own `create` operation, which uses the identical sequencing for the identical reason.
- **Type ID updates carry the previous type script unchanged.** The caller must pass `prevTypeScript` (read from the previous live cell) alongside `prevOutPoint`; `buildAnchorTxWithTypeId` throws if one is given without the other rather than silently computing a fresh (and wrong) Type ID for an update.
- **Placeholder sighash witness before fee completion.** `Transaction.completeFeeBy` sizes the fee from the tx's current byte length; a `SignerCkbScriptReadonly` never attaches a witness, so without help the estimate undershoots by exactly one signature's worth of bytes and real broadcasts get rejected by the node's minimum relay fee rate once the caller's actual signer adds its witness later. All three builders call `tx.prepareSighashAllWitness(lock, 65, client)` (a 65-byte placeholder — CCC's own convention for a secp256k1/JoyID sighash-all signature) before `completeFeeBy`; the real signer overwrites the same witness position with its actual signature, so final size matches the estimate.
- **Default fee rate: 2000 shannons/KB, not CKB's bare 1000 minimum.** Confirmed against a real offckb devnet (CKB 0.207.0): even with the witness placeholder above, broadcasting at the exact minimum rate was rejected by a few dozen shannons — the node's actual fee accounting isn't quite pure byte-count. Doubling the floor costs nothing meaningful and stops relying on exact agreement between CCC's size estimate and a given node's metering.
- **Devnet system scripts are not hardcoded.** A devnet's Secp256k1Blake160 (and other non-native scripts) are deployed in its own genesis block, so their cell-dep out points differ per devnet — unlike Type ID, which is a native VM feature with a fixed code hash on every network. `makeClient("devnet")` reads an optional `VERICELL_DEVNET_SCRIPTS_FILE` (JSON from `offckb system-scripts --export-style ccc`) and passes it as the CCC client's `scripts` override; without it, devnet falls back to testnet's scripts, which is wrong for anything that needs to sign (confirmed live: `buildAnchorTx` → `signer.sendTransaction` fails to resolve the cell dep without this file). Verified end-to-end against a live offckb devnet (anchor, supersede, withdraw, and the Type ID variant of the same flow all broadcast and confirm correctly).
- **offckb devnet RPC defaults to `http://127.0.0.1:28114`**, not 8114 — offckb's own RPC proxy listens there by default. Overridable via `VERICELL_DEVNET_RPC_URL`.
- **`FakeClient` (unit tests only, not exported from the package's public barrel).** A from-scratch `ccc.Client` subclass with an in-memory cell/tx/header store, rather than mocking `ClientJsonRpc`'s transport layer — implementing the handful of abstract methods our builders actually exercise was more legible than reverse-engineering which JSON-RPC calls CCC's higher-level helpers happen to make.

## Phase 3 — packages/api: database and indexer

- **Two bugs found via this phase's own offckb integration test (three sequential anchors from one funded devnet account), both real for any wallet anchoring more than one project — Phase 5's custodial mode anchors many projects from one `SERVICE_PRIVATE_KEY` wallet, so this was live production risk, not just a test artifact:**
  1. **Indexer: `input.previousOutput.index` was ignored when detecting consumption.** Every builder in `chain` places the proof cell at output index 0; a *later*, unrelated anchor from the same wallet routinely spends an *earlier* anchor tx's index-1 change cell as plain capacity funding — a completely ordinary, harmless on-chain event. `processBlock`'s consumption check matched only on `previousOutput.txHash`, so spending that change cell was indistinguishable from spending the proof cell itself: `project1`/`project2` came back `active = 0`/`status = 'consumed'` in the DB while their cells were still verifiably live on-chain (confirmed via direct `getCellLive`). Fixed by skipping any input whose `previousOutput.index !== 0` before treating it as a candidate consumption (`indexer/process.ts`).
  2. **`packages/chain`: capacity collection had no filter, so it could also pick up a *different* live proof cell (not just a change cell) as funding**, e.g. a wallet whose only spendable capacity happens to be an old proof cell. Independent of bug 1 (which is about mis-*reading* a legitimate spend; this is about *making* an illegitimate one) — both needed fixing for the integration test to reflect correct behavior end-to-end. Fixed with a shared `PURE_CAPACITY_FILTER` (`{ outputDataLenRange: [0, 1] }`, `chain/src/filters.ts`) passed to every capacity-collecting call in `anchor.ts` — capacity funding now only ever touches empty-data cells.
- **`versions.consumed_at_block` (not in TECHNICAL.md §6 verbatim).** Reorg rollback must precisely undo a `status = 'consumed'` flip when the *consuming* transaction is the one being rolled back, while leaving a version alone if it was validly consumed at or before the fork point (a real supersede/withdraw the reorg doesn't touch). The schema as given has no way to tell these apart after the fact — rolling back only rows with `block_number > forkPoint` deletes the successor's own row but leaves the predecessor's `status` stuck on `consumed`. Added a nullable `consumed_at_block BIGINT` column to `versions`, set whenever a version is marked consumed; rollback resets `status`/`consumed_at_block` for exactly the rows whose `consumed_at_block` is past the fork point. Documented here rather than silently deviating from "verbatim" because it's additive (no listed column changed) and load-bearing for correct reorg behavior, which the phase brief itself calls for ("delete/recompute rows").
- **Full-chain block scan, not a filtered indexer RPC query.** The indexer fetches every block from the cursor to tip and inspects every transaction's outputs/inputs, rather than querying by the VeriCell type script (TECHNICAL.md's own roadmap item 1 lists "CKB indexer RPC, filtered by VeriCell type script" as a future optimization, not a Phase 3 requirement — and a fixed Type ID *script* isn't unique to VeriCell per project the way its *args* are, so a script-only filter can't narrow past "some Type ID cell" anyway). Acceptable at devnet/testnet scale for this phase; revisit if/when indexing mainnet from genesis becomes a real cost.
- **Reorg rollback depth is a fixed `N` (default 6 blocks), not a walk-back to the exact fork point.** `sync_state` only stores the *last* block's hash (per TECHNICAL.md §6, verbatim), not a history of recent hashes, so there's no way to binary-search the true fork point from persisted state alone. Matches the phase brief's own wording ("roll back N blocks"). Self-correcting: if the reorg is deeper than `N`, the next poll iteration detects the still-mismatched parent hash and rolls back another `N`, bounded below by block 0. `pollOnce` also re-validates the last-recorded block's hash *before* trying to fetch anything beyond it, so a same-height or shorter reorg (chain length never exceeds the old tip) is still caught — the parent-hash check alone only fires while fetching blocks past the cursor.
- **`IndexerClient` is a narrow structural interface** (`getTip`, `getBlockByNumber`, `getKnownScript`, `addressPrefix`), not `ccc.Client` itself. A real client satisfies it with no adapter; tests pass a plain in-memory `FakeChainClient` (`indexer/fakeChainClient.ts`, unit tests only, not exported — same rationale as `chain`'s own `FakeClient`) instead of subclassing `ccc.Client`'s full abstract surface. `projects.ckb_address` is rendered via `new ccc.Address(Script.from(lock), addressPrefix)` rather than the `Address.fromScript(script, client)` static helper, specifically so the indexer only ever needs that one string off the client, not a full `Client` instance.
- **DB path:** `DB_PATH` env, when set, is used verbatim (trusted to already be network-scoped by whoever set it — matches Phase 10's `.env.example`); otherwise defaults to `data/vericell.<network>.sqlite`, satisfying the network-scoping rule with no configuration required. `DB_PATH=:memory:` is how tests get an isolated, disposable DB.

## Phase 4 — packages/api: public REST endpoints

- **JSON field casing: snake_case throughout the public API**, matching the §6 DB column names directly and TECHNICAL.md §7.1's own literal `/verify` shape (`{ found, live, project, version, block_time, path }`). Chosen over camelCase specifically so that one endpoint's spec-mandated field names don't clash stylistically with the rest of the API's freely-chosen ones.
- **`/versions/{txHash}` always calls the chain, regardless of `source`.** TECHNICAL.md §6's `versions` table never persists the raw manifest (only `project_sha256`/`merkle_root`), so an indexed row alone can't answer "manifest, chain status, block number & timestamp, owner" (§7.1's description of this endpoint). `source: "index"` vs `"chain"` therefore doesn't mean "chain lookup skipped" — it means whether a DB row also exists to supply `version_no`/`status`/`prev_tx_hash` (indexed) or those are derived purely from the RPC response (`live ? "committed" : "consumed"`, unindexed). If the chain call itself fails, an indexed row still answers with `manifest: null` rather than a hard error; an unindexed lookup with no DB row and a failed chain call surfaces as 502, since there is nothing else to answer from.
- **`hash` filters (`/projects?hash=`, `/hashes/{sha256}`) match either a whole-project hash (`versions.project_sha256`) or a per-file hash (`hashes.sha256`).** TECHNICAL.md describes the `/projects` filter only as "any SHA-256" and `/hashes` only as "backward search" without pinning down which hash kind — matching both is the more useful default and costs one extra `EXISTS`/`UNION` clause.
- **`fetchProof`/`getTip` are injected into `buildServer()`** (`server/chainLookup.ts`), defaulting to real `chain.makeClient()`-backed implementations built lazily on first use. Every route test injects a fake instead — mirrors Phase 3's `IndexerClient` pattern (narrow injectable interface, not a full `ccc.Client`) and is what makes the `inject()` test suite fast and offline.
- **Routes are registered inside a child `app.register(async (instance) => {...})`, not directly on the root `app`.** `@fastify/swagger`'s `onRoute` hook is attached during its own (asynchronous, avvio-queued) plugin boot; routes declared directly on `app` immediately after `app.register(swagger, ...)` run before that hook exists and are silently omitted from the generated OpenAPI spec. Wrapping routes in their own `register()` call puts them later in avvio's FIFO boot queue, after swagger has already run.
- **Fastify v5's `logger` vs `loggerInstance` split.** `buildServer`'s `logger` option is boolean-only (Fastify's own request logging); a pre-built logger (e.g. `pino()` in `server/run.ts`) must go through the separate `loggerInstance` option — passing a real logger instance as `logger` throws `FST_ERR_LOG_INVALID_LOGGER_CONFIG` at startup. Caught by manually smoke-testing the built `server/run.ts` binary over real HTTP (the `inject()` test suite never exercises this path, since it never passes a logger).
- **`/health`'s `indexer.lag`/`tip` never fail the request.** A chain RPC error while computing `tip` sets `chain_reachable: false` and `lag: null` but still returns 200 — the endpoint's own liveness (the API process is up and can query its DB) is independent of whether the upstream chain node is reachable.
- **fastify-type-provider-zod for validation + OpenAPI generation**, rather than hand-rolling JSON Schema or a manual zod-parse-then-format-error step per route. `hasZodFastifySchemaValidationErrors`/`isResponseSerializationError` type guards feed directly into the RFC 9457 error handler; `jsonSchemaTransform` feeds `@fastify/swagger` so route `params`/`querystring` zod schemas double as the OpenAPI spec with no separate schema definitions to keep in sync.
- **Rate-limit errors are formatted by the same central `setErrorHandler`**, not `@fastify/rate-limit`'s own `errorResponseBuilder`. `@fastify/rate-limit` sends its 429 via `reply.send(error)`, which still flows through Fastify's configured error handler (confirmed against the plugin's own docs/behavior) — so a single `statusCode === 429` branch there is enough to get problem+json on rate-limited responses too, no plugin-specific configuration needed.

## Phase 5 — packages/api: authenticated write endpoints

- **`makeClient`'s `VERICELL_RPC_URL`** overrides the RPC endpoint for *any* network (self-hosted node, private gateway). Precedence: `VERICELL_RPC_URL` > `VERICELL_DEVNET_RPC_URL` (devnet only) > network default. Passing `url: undefined` to `ClientPublicMainnet`/`ClientPublicTestnet` falls through to their own built-in default, so this is additive — no behavior change when unset.
- **Indexer shutdown: a per-`start()` `AbortController`, not the existing `stopped` boolean, gates mid-catch-up interruption.** `pollOnce()`'s block-fetch loop now checks `this.abortController.signal.aborted` every iteration (not just between `pollOnce()` calls), and the idle poll-interval sleep uses `node:timers/promises`' `setTimeout(..., { signal })` so `stop()` interrupts it immediately instead of waiting out the full interval. `stopped` alone couldn't gate the loop directly: it defaults to `true` before `start()` is ever called, and tests call `pollOnce()` standalone (see its own doc comment) expecting a full catch-up regardless — the constructor's default (never-aborted) `AbortController` preserves that, while `start()`/`stop()` get a fresh one per cycle.
- **`INDEXER_START_BLOCK` (env var read in `indexer/run.ts`), not `VERICELL_INDEXER_START_BLOCK`.** Follows the un-prefixed convention of this phase's other operational env vars (`DB_PATH`, `PORT`, `POLL_INTERVAL_MS`) rather than the `VERICELL_`-prefixed convention used by the network-flag family (`VERICELL_NETWORK`, `VERICELL_RPC_URL`) — matches `ClaudeCodeInstruction.md`'s literal naming for this phase.
- **New anchors default to Type ID**, unconditionally, for both non-custodial `/proofs/prepare` (no `prev_tx_hash`) and custodial `POST /proofs` — TECHNICAL.md §5 calls Type ID "Production," and the write API has no reason to ever mint a new v1 (no-Type-ID) project. Versioning an *existing* project (`prev_tx_hash` / `POST /proofs/{unid}/versions`) detects Type ID dynamically from the previous live cell instead (`buildAnchorTxWithTypeId` if present, else `buildAnchorTx`), so a legacy v1 project anchored before this phase can still be versioned through the write API without migration.
- **`/proofs/prepare` computes `project_sha256`/`merkle_root`/`count` server-side from a caller-supplied `files` list**, never trusting a caller-supplied hash of the file list — the manifest draft schema (`writeSchemas.ts`) makes `files` required (unlike the on-chain `ManifestSchema`, where it's optional for compact mode) specifically so the server always has enough to recompute these itself, per the phase brief's "computed project_sha256."
- **Tx wire format between `/proofs/prepare` and `/proofs/submit`: `JSON.parse(ccc.stringify(tx))`, round-tripped back via `ccc.Transaction.from`.** `ccc.stringify` is CCC's own bigint-safe serialization (capacities etc. become hex strings); empirically verified this round-trips to a `Transaction` with an identical `.hash()`, including after `signTransaction` attaches a real witness over the placeholder `reserveSighashWitness` left. Chosen over a hand-rolled schema so the wire format tracks whatever `Transaction`/`TransactionLike` actually contains, with no separate encode/decode to keep in sync.
- **Pending-row insertion (`insertPendingVersion`, `server/writeQueries.ts`) advances `projects.live_tx_hash`/`live_index` optimistically on *every* submit — including a new version of an already-existing project, not just a brand-new one.** Originally scoped narrower (`ON CONFLICT DO NOTHING`, leaving the old version "live" until indexed) on the theory that a not-yet-confirmed tx shouldn't be optimistically advertised; that shipped with a real bug, caught only by the offckb integration suite (not the `FakeClient`-based unit tests — see below): chaining custodial anchor → new version → withdraw in quick succession resolved `live_tx_hash` to the *previous* (by-then-already-consumed) cell, so the withdraw built a tx against a dead outpoint and the real node rejected it with `TransactionFailedToResolve`. `FakeClient`'s old `sendTransactionNoCache` had no dead-input validation at all, so the unit test's equivalent flow "passed" without ever exercising the bug. Fixed both: `insertPendingVersion` now does `ON CONFLICT DO UPDATE` (mirrors `indexer/process.ts`'s own `upsertCandidate` shape) unconditionally, and `FakeClient.sendTransactionNoCache` now throws on any input whose outpoint isn't currently live, matching real node behavior — so this class of bug is unit-testable going forward. The accepted trade-off (a pending tx that never confirms leaves `live_tx_hash` pointing at a technically-not-live cell) was already implicit for brand-new projects; reconciling a stuck/dropped pending tx is out of scope for this phase.
- **`projects.created_at` on a `versionNo === 1` conflict is overwritten with the indexed block time; any other version number leaves it alone** (`indexer/process.ts`'s `upsertCandidate`, `CASE WHEN @versionNo = 1 THEN excluded.created_at ELSE projects.created_at END`). Needed once pending-row insertion could pre-create a project's row before the indexer ever sees it: the pending row's `created_at` is a submit-time guess, and only the genesis version's indexed write should ever get to correct it — a later version hitting the same `unid` on conflict (the ordinary "new version indexed" case Phase 3 already exercised) must never touch it.
- **`api_keys` rows carry no per-caller identity beyond the key hash itself** — no `owner`/`account` column, no notion of "which caller anchored which project." Custodial ownership is instead checked directly against the *service wallet's own lock* (`prevCell.cellOutput.lock.eq(serviceLock)`) before a version/withdraw is allowed, which is the only ownership fact that's actually true on-chain in custodial mode: every custodial anchor shares the one service-wallet lock, so any valid API key can act on any custodial project. TECHNICAL.md doesn't specify per-key multi-tenant ACLs for custodial writes; adding one would be scope creep beyond "automation anchoring."
- **Idempotency scope: `(api_keys.key_hash, Idempotency-Key)`, keyed additionally by method + registered route pattern (`req.routeOptions.url`, not the literal request URL).** A stored response whose method/path doesn't match the replay 409s (`ConflictError`) rather than silently replaying the wrong response — a caller reusing a key across two different endpoints is almost certainly a bug on their end, not an intentional replay. `POST /api/v1/keys` (no caller API key yet — that's what it mints) uses the fixed scope `"admin"` instead, since every admin caller shares the one `ADMIN_TOKEN`.
- **Per-key rate limiting is a second, independent `@fastify/rate-limit` registration**, scoped to the `/proofs*` child plugin (nested inside the route-registration child plugin, itself already required for the swagger `onRoute` timing reason from Phase 4). Both `keyGenerator` and `max` read the bearer token straight off the request header and do a synchronous `better-sqlite3` lookup — no separate preHandler coordination needed, since rate-limiting's `onRequest` hook naturally runs *before* `requireApiKey`'s `preHandler`: an unrecognized/missing key is capped at a low fixed ceiling (5/min) by the rate limiter and then 401s in the preHandler right after, rather than ever consuming a real per-key limit it doesn't have.
- **`FakeClient.getKnownScript` fixtures extended (`Secp256k1Blake160`, `AnyoneCanPay`)** so a real `ccc.SignerCkbPrivateKey` can be used against it end-to-end (`getRecommendedAddressObj`, `signTransaction`) — needed for `api`'s route-level unit tests to exercise a genuine prepare → sign → submit round trip without a live devnet. Both are dummy fixture values in the same spirit as the existing `NervosDao` one: never matched by any fixture cell, only resolved so CCC's own signer-side code (which unconditionally checks for related-script code hashes) doesn't throw.
- **`chain`'s `FakeClient` is now exported from the package's public barrel** (previously deliberately unexported, per this file's Phase 2 entry) — `api`'s Phase 5 route tests are the first cross-package consumer that legitimately needs a working fake `ccc.Client`, and duplicating it inside `api` would drift from the real one's behavior (as the dead-outpoint fix above demonstrates, that behavior matters).
- **Custodial mode's mainnet guard (`resolveCustodialEnabled`, `server/chainClient.ts`) lives in this phase, not deferred to Phase 10.** TECHNICAL.md §9 / the global rules describe it as a property of custodial mode itself ("refuses to start custodial mode unless `MAINNET_CONFIRM=1`"), and Phase 5 is what actually implements custodial mode — enforcing it here means there's never a window where custodial mode exists without the safeguard. Implemented as "silently don't enable custodial routes' underlying capability, with a logged warning" rather than crashing the whole server, since a misconfigured custodial flag shouldn't take down the public read API too.

## Phase 6 — Webhooks

- **`webhooks.secret` is stored in cleartext, not hashed.** Unlike `api_keys.key_hash` (only ever compared for equality against a caller-supplied value), the webhook secret has to be read back at delivery time to *compute* the outgoing HMAC — a one-way hash can't do that. Added as a new column via `0003_webhooks.sql` rather than deviating from the §6 `webhooks` table verbatim. Returned once in the `POST /webhooks` response body, like an API key, but — unlike an API key — the receiver needs to retain it indefinitely to verify every future delivery, not just the first.
- **Webhook scope: `unid IS NULL` = every project; `unid = X` = only that project — independent of which API key registered it.** TECHNICAL.md §6's own column comment says exactly this ("NULL = all projects of this key"), but Phase 5 already established that `api_keys` carries no per-caller project ownership (any key can act on any custodial project — see that phase's entry). Tracking "projects belonging to a key" would be new scope beyond what any endpoint needs; "all projects, globally" is the simpler and only self-consistent reading once ownership-by-key doesn't exist. "Key-scoped" in the phase brief is satisfied by `DELETE` instead: only the key that registered a webhook may remove it (`webhooks.key_hash`), independent of which projects' events it watches.
- **Event semantics.** `committed` fires once per version, the first time the indexer sees its tx on-chain (tracked via a pre-upsert `SELECT status` in `upsertCandidate` — never re-fires on a later re-index of an already-committed row, though a reorg-then-replay can refire it once; treated as an acceptable edge case, not handled specially). `consumed` fires on the version whose cell was spent, whether by a successor or a withdraw. `superseded` fires additionally (alongside `consumed`), scoped to the project, only when a same-tx successor exists — TECHNICAL.md §7.3 glosses it as "(new version live)", and its payload carries both the just-superseded version's `tx_hash` and the new live version's `successor_tx_hash` so a receiver doesn't have to correlate two separate deliveries to learn the new live tx hash.
- **Deliveries enqueued inside the same `db.transaction()` as the triggering write** (`indexer/process.ts` calls `enqueueWebhookDeliveries` directly, no callback indirection) — a delivery is never queued for a state change that a reorg then rolls back within the same `processBlock` call. A reorg rolling back an *already-processed* block (a separate, later `pollOnce()` iteration) does not retract or cancel any `webhook_deliveries` rows that block's processing already enqueued/sent — out of scope for this phase; the same trade-off already accepted for `versions`/`projects` rows becoming stale is analogous, and an already-sent HTTP delivery can't be unsent regardless.
- **Dispatcher runs once per `pollOnce()` call, after the block catch-up loop, not once per block.** Simpler than interleaving HTTP calls between synchronous block-processing transactions, and it still runs on every poll interval even when zero new blocks were indexed — the only way a due retry (scheduled on a previous poll, no new chain activity since) ever gets re-attempted.
- **Retry backoff: `1s, 2s, 4s, 8s, ... capped at 5 minutes`, `maxAttempts = 5` counted as total attempts including the first** (not 5 retries *after* an initial attempt, i.e. 6 total). ClaudeCodeInstruction.md Phase 6 says "retries (max 5)"; total-attempts-of-5 is the simpler reading and matches common webhook-provider conventions (Stripe, GitHub) closely enough without inventing a distinction the brief doesn't ask for.
- **`Indexer` takes injectable `webhookFetch`/`now`/`webhookMaxAttempts`**, mirroring the existing injectable-dependency pattern (`IndexerClient`, `fetchProof`/`getTip` in `server/chainLookup.ts`) — lets the webhook delivery test suite point at a real local HTTP receiver while still controlling retry timing deterministically via a fake clock, with no real `sleep` in any test.
- **SSRF guard (`webhooks/guard.ts`) denies by IP range after resolving the hostname (or reading it directly if already a literal IP), not by a hostname blocklist.** A blocklist of hostnames (`localhost`, etc.) is trivially bypassed by any attacker-controlled DNS record; resolving first and checking the actual address is what closes that gap. Re-checked immediately before every delivery attempt (not just at registration) as cheap defense-in-depth against DNS rebinding between registration and a much-later retry. `WEBHOOK_ALLOW_PRIVATE_NETWORKS=1` is the escape hatch for local testing (this phase's own test suite posts to a `127.0.0.1` receiver) and must never be set in a real deployment.
- **IPv6 private-range detection is a hand-rolled parser** (`expandIpv6Groups`/`isPrivateIpv6`), not a dependency — Node has no built-in "is this IPv6 address private" check, and pulling in a package for `::1`/`fc00::/7`/`fe80::/10`/`::ffff:0:0/96` (the ranges that actually matter for an SSRF guard) was judged not worth a new dependency for this phase.

## Fix — offckb suite contention (pre-Phase-7)

- **`packages/api`'s two offckb suites (`indexer/offckb.integration.test.ts`, `server/proofs.offckb.integration.test.ts`) build transactions against the same funded devnet account.** Vitest's default file parallelism ran both in separate workers at once; each independently selects live capacity cells owned by that one account via `completeInputsByCapacity`, so one suite's not-yet-confirmed spend could still look selectable to the other, and the node rejected the resulting tx with `502 TransactionFailedToResolve` (confirmed: the proofs suite failed only when run alongside the indexer suite, never alone). Fixed at the config level — `packages/api/vitest.config.ts` sets `fileParallelism: false` whenever `OFFCKB=1` — rather than adding locking inside the suites themselves, since a single shared devnet account makes "run one offckb file at a time" the actual invariant that needs to hold, and non-offckb `pnpm --filter api test` runs are untouched (still fully parallel).
- **Per-suite devnet account override, additive on top of serialization.** `VERICELL_OFFCKB_PRIVATE_KEY_INDEXER` / `VERICELL_OFFCKB_PRIVATE_KEY_PROOFS` let each api offckb suite use its own funded devnet account when one is available, falling back to the shared `VERICELL_OFFCKB_PRIVATE_KEY` otherwise. Not required for correctness once suites are serialized, but removes the contention risk entirely for anyone who has more than one funded devnet account handy, and costs nothing when unset.
- **Timeouts raised generously across every offckb suite** (`packages/chain`'s new `vitest.config.ts`, `packages/api`'s existing one, and the explicit per-`it`/`beforeAll` timeouts in all three offckb test files) — serializing `api`'s two suites roughly doubles the wall-clock time of `test:offckb` on a slow devnet, and CI/sandboxed environments have already shown real cell-resolution and transaction-confirmation latency well past the previous defaults.

## Phase 7 — `packages/web`: upgrade the SPA

- **Search always queries both the API and the local registry, merges by tx hash (API wins), and renders provenance badges** ("global index" vs "this device") rather than switching to API-only once one is configured. `apiSearch()` never throws — a failed/absent API call returns `{ hits: [], apiDown: true }` and the UI falls back to local-only results with a visible "global index unavailable" note, satisfying "work with the API up and gracefully degrade without it" without a separate offline/online mode to maintain.
- **`GET /projects?hash=` covers the "paste a hash" search case, not a client call to `/verify/{sha256}`.** Both endpoints match the same rows per this file's Phase 4 entry (project *or* file hash); `/projects` returns full project rows the search UI can render as clickable result cards (title, address, live status), where `/verify`'s convenience-verdict shape is built for a single-answer consumer (the CLI), not a list. Avoids standing up a second client code path for no behavioral gain.
- **List-search rows (`GET /projects`) don't carry `project_sha256`** (that lives on `versions`, not `projects` — see Phase 4's schema), so a search-result card's fingerprint strip is filled in lazily via a follow-up `GET /projects/{unid}` per visible card, while the LIVE/consumed badge renders immediately from the list row's own `active` field (no chain round-trip needed for API-sourced results, unlike the pre-Phase-7 behavior every result still uses for local-registry-only rows).
- **Project detail (`GET /projects/{unid}`) is tried first; a direct on-chain lookup of just the clicked version is the fallback** — used both when the API is unreachable and when a project genuinely isn't indexed yet (freshly anchored, not yet caught up by the indexer). The version *timeline* only ever comes from the API (the chain alone has no way to enumerate a project's other versions without a full scan); the fallback view is deliberately the same single-version detail page Phase-7-era v1 always showed, not a degraded imitation of the timeline.
- **Hashing/Merkle/manifest-building code is deleted from `packages/web` in favor of importing `core` directly** — `sha256Hex`, `projectHash`, `merkleRoot`, `encodeManifest`, `estimateCellCost` all come from the same package the API and chain layers use, so a manifest built in the browser is byte-identical to one built server-side by construction, not by convention. The only adapter kept is `toFileEntries()`, translating the UI's `{p, h}` file-entry shape (matching the on-chain manifest's own `{p, h}` field names) to `core`'s `{path, hash}` `FileEntry` type.
- **The network `<select>` is gone; `NETWORK`/`EXPLORER_URL`/`isMainnet()` from `core/network.ts` drive a read-only top-bar badge instead.** A web *build* now targets exactly one network (baked in via `VITE_VERICELL_NETWORK`), matching every other package — letting users flip networks client-side was a v1-only affordance that predates the network flag existing at all. `makeChainClient()` is web's own thin `NETWORK -> ccc.Client` mapping (mirroring `chain`'s `makeClient`, not imported from it — `chain/client.ts` reads `node:fs` for devnet's system-scripts file, which doesn't bundle for a browser).
- **The "API & automation" section's curl examples and docs link are filled in by JS at init (`renderApiSection`), not hardcoded in `index.html`.** The API base is a runtime value of the *build* (`VITE_API_URL`), so the same markup has to render sensibly both when it's configured (real, clickable docs link) and when it isn't (a placeholder base URL and a note telling the operator what env var to set) — static HTML can't express that conditionally.

## Phase 8 — `packages/cli`

- **Non-custodial `withdraw` has no API counterpart** — TECHNICAL.md §7.2-B's `DELETE /proofs/{unid}` is custodial-only (signed by the service wallet), and Phase 5 never added a `prepare`/`submit` pair for withdrawal. Rather than extend the API in a phase that isn't supposed to touch it, `vericell withdraw --mode non-custodial` builds and broadcasts the withdraw transaction directly against the chain (`chain.buildWithdrawTx`, signed locally), the same way `anchor --mode non-custodial` never sends a private key to the server — it only reads the live cell's location from the public `GET /projects/{unid}`. `--mode` defaults to `custodial` for this command specifically (the only mode with a real API route); `anchor` has no default and requires `--mode` explicitly, matching the instruction file's literal signature.
- **`vericell hash --compact` manifests can't be fed back into `vericell anchor`.** TECHNICAL.md §3's compact mode omits `files` from the on-chain manifest, but the API's `ManifestDraftSchema` (`POST /proofs/prepare`/`/proofs`) always requires `files` — the server needs the real hashes to independently recompute `project_sha256`/`merkle_root` even when it won't embed them on-chain. A compact manifest.json has nowhere to recover that list from (the CLI only has the manifest file at anchor time, not the original directory), so `anchor` fails fast with a clear message pointing back at re-running `hash` without `--compact`, rather than silently producing a manifest with an empty file list.
- **`.gitignore` handling is single-root, not git's full cascading semantics.** `walkPaths` reads only the `.gitignore` at a walked directory's own root (via the `ignore` package) and always skips `.git`; nested `.gitignore` files elsewhere in the tree are not merged. Covers the flow this phase actually targets (hashing a release/build output directory or a repo checkout) without reimplementing git's directory-scoped ignore-file cascade for a CLI convenience feature. The `.gitignore` file itself is still hashed like any other tracked file, matching real git behavior (nothing tells git — or this walker — to ignore `.gitignore`).
- **The CLI e2e offckb suite (`cli.offckb.integration.test.ts`) sets `VERICELL_NETWORK=devnet` explicitly in the spawned subprocess's environment, not just ambiently inherited.** Unlike `chain`'s/`api`'s offckb suites, which call `makeClient("devnet")` directly in-process, the CLI resolves its network from `core`'s env-based `NETWORK` default (`packages/core/src/network.ts`), which is `testnet` when unset — without the explicit override, the CLI subprocess would silently sign against public testnet infrastructure (wrong cell dep out points for a local devnet lock) while the in-process test API still talked to the real local devnet, producing a confusing `TransactionFailedToResolve` unrelated to the payer account. Discovered by comparing a manual same-flow reproduction (which set the network correctly and succeeded) against the failing automated run.
- **The e2e suite rebuilds the CLI (`pnpm --filter cli build`) at the start of its own `beforeAll`** rather than assuming `dist/` is already fresh — it's gated behind `OFFCKB=1` like every other integration suite in this repo, so the extra build cost only applies to an already-slow, already-gated run; `pnpm test` (the always-green suite) never depends on a prebuilt `dist/`.

## Phase 9 — Hardening & full test pass

- **`migrations/0004_perf_indexes.sql` adds `idx_versions_project_sha256`.** `getHashMatches` (`server/queries.ts`, backing both `GET /hashes/{sha256}` and `GET /verify/{sha256}`) UNIONs a lookup by `hashes.sha256` (already indexed since `0001_init.sql`) with one by `versions.project_sha256`, which had no index — `EXPLAIN QUERY PLAN` against a 10k-project/200k-hash fixture confirmed that half of the query fell back to a full table scan of `versions` before this migration. A new append-only migration, not an edit to `0001_init.sql`, per the migration runner's own "applied at most once, in filename order" contract — editing an already-applied file wouldn't retroactively add the index to any DB that already ran `0001_init.sql`.
- **The load-sanity suite (`server/hashLookup.loadSanity.test.ts`) warms both routes once in `beforeAll` before asserting the <50ms budget.** Fastify compiles a route's zod validator/serializer lazily on its first request — confirmed via direct `EXPLAIN QUERY PLAN` and raw `better-sqlite3` timing that the indexed query itself runs in well under 1ms against the 200k-hash fixture, while the *first* `app.inject()` call in the unwarmed suite took ~260ms, all Fastify/route-compilation cold start unrelated to the query. Real load-test methodology excludes cold start from steady-state measurements for the same reason; asserting on the unwarmed first call would have measured "how slow is compiling a Fastify route the first time," not "is this query indexed."
- **Webhook delivery sets `redirect: "manual"` and treats any 3xx as a failed delivery, rather than following redirects.** The SSRF guard (`webhooks/guard.ts`) only ever validated the registered `webhooks.url`'s own DNS resolution; `fetch`'s default `redirect: "follow"` meant a receiver that later 30x-redirected the actual delivery POST could point the real request at a private address without that address ever passing the guard. A receiver that legitimately relocates must be re-registered at its new URL — silently following a redirect through unvalidated territory isn't an acceptable trade-off for a guard whose entire purpose is validating the request's destination.
- **`warnIfMainnet` (`packages/api/src/mainnetWarning.ts`) is a small shared helper called from both `server/run.ts` and `indexer/run.ts`, not inlined in each.** The global rule's "logs a prominent warning" on mainnet startup is unconditional (independent of `resolveCustodialEnabled`'s custodial-specific refusal, which already existed) and applies equally to both standalone entrypoints; a shared function keeps the two call sites from drifting in wording or in what counts as "mainnet."
- **`idempotency_keys.response_body` for `POST /api/v1/keys` retains the raw minted key in cleartext, unchanged from Phase 5.** True idempotent replay requires storing the exact original response verbatim, and that response *is* the one-time secret — there's no way to satisfy both "replay returns the identical response" and "never persist the secret." Judged an accepted trade-off rather than a fix: `webhooks.secret` is already stored in cleartext by deliberate design (this file's Phase 6 entry), so the DB file already has to be treated as secret-bearing; this doesn't introduce a new class of exposure, only extends an already-accepted one. See `docs/SECURITY.md` §1.
- **`packages/cli` had zero unit-level test coverage before this phase** — its only test file was the `OFFCKB=1`-gated e2e suite from Phase 8, which `pnpm test` skips entirely, so `v8`'s coverage tool reported 0% across the whole package even though the commands were exercised black-box by that suite when run. Closed with a full pass: `lib/*.test.ts` (100% stmts — `walk`, `apiClient`, `manifestDraft`, `signer`, all pure/mockable with no real network) and `commands/*.test.ts` (92.8% stmts). `anchor.test.ts`/`withdraw.test.ts` mock the `chain` module's `makeClient` export (`vi.mock("chain", ...)`, returning a shared `FakeClient`) so the non-custodial sign-and-broadcast path runs against a real in-memory chain client instead of a real network — the same technique `packages/api`'s own route tests already use via `buildServer({ chainClient })`, applied here because `runAnchor`/`runWithdraw` call `makeClient()` internally rather than accepting it as a parameter. `index.ts` (commander wiring) is the one file left at 0% — reasonably left to the offckb e2e suite's black-box coverage, since unit-testing argument-parsing wiring in-process mostly duplicates what that suite already proves end-to-end.
- **`packages/api` coverage work targeted authorization and error-branch gaps over cosmetic line-count, notably: `resolvePrevVersion`'s entire non-custodial "anchor a new version" path (0% covered — no test, unit or offckb, had ever set `prev_tx_hash` on `/proofs/prepare`), the custodial `.../versions` and `DELETE /proofs/{unid}` routes' service-wallet ownership check (403 branch — previously untested, meaning a broken ownership comparison would have shipped silently), and `resolveCustodialEnabled`'s mainnet refusal (the exact function backing the mainnet safety guard reviewed in `docs/SECURITY.md`, previously with no dedicated test at all).** Chosen deliberately over chasing the remaining small branch-coverage gaps in ternary tie-breaks and process-entrypoint bootstrap code (`server/run.ts`, `indexer/run.ts`, `chainClient.ts`'s default network-touching factories), which are lower-risk and, for the entrypoints, conventionally hard to unit test without real network/process side effects.
- **`--api` normalization lives in `ApiClient`'s constructor (`normalizeApiBaseUrl`), applied once, rather than in each command or in `index.ts`'s option parsing.** Every command builds an `ApiClient` from the raw `--api` value, so normalizing there covers `hash`/`anchor`/`verify`/`status`/`withdraw` uniformly from one implementation and one test file — a bare origin (`http://host:3000`) and a full base URL (`http://host:3000/api/v1`) now behave identically everywhere, found missing in manual testing of the Phase 8 CLI. The check is a case-insensitive suffix match on `/api/v1` (not a full URL-structure parse) — deliberately simple, matching the one real ambiguity this needs to resolve (bare origin vs. already-prefixed) rather than trying to validate arbitrary reverse-proxy path setups.
- **A non-JSON response body is now a distinct, hinted `ApiRequestError` rather than an uncaught `SyntaxError` from `JSON.parse`.** Manual testing showed this is exactly what happens when `--api` points at the wrong thing (the web app's dev server, a reverse proxy's default page, a typo'd port) — the raw `JSON.parse` crash gave no indication why, so the CLI now recognizes "response wasn't JSON" as its own case and suggests checking `--api` against the documented `/api/v1` base URL. A network-level failure (`fetch()` itself rejecting — DNS, connection refused) gets the same treatment with a distinct message, since the *cause* an operator should look for differs (server unreachable at all, vs. reachable but serving the wrong thing).

## Phase 10a — dual-network backend

- **Route mounting: the same registration functions, called three times inside three encapsulated Fastify child scopes, not a `:network` route parameter.** Every route file's path was stripped of its hardcoded `/api/v1` prefix (e.g. `/projects`, `/proofs/:unid`); `buildServer` then registers that same set once per configured network under `prefix: "/api/v1/<network>"`, and once more under bare `prefix: "/api/v1"` for the default-network alias. Each mount's `app.decorate("db", ...)`/`("network", ...)`/etc. only shadow that mount's own child scope (verified empirically: Fastify allows a child to shadow a decorator its parent already has, scoping the override to that child and its own descendants) — so sibling mounts can never see each other's database or chain client, with zero changes to any route handler (`app.db` reads exactly as before everywhere). A `:network` path-param route dispatching internally per request was the alternative; rejected because it would have meant threading a per-request network lookup through every handler/query function that currently reads `app.db` directly, for no behavioral gain.
- **`BuildServerOptions`'s pre-10a single-network fields (`db`, `network`, `fetchProof`, `getTip`, `chainClient`, `custodialEnabled`, `custodialSigner`) become the `networks[defaultNetwork]` entry when no explicit `networks` map (or no entry for that network) is given.** This is the entire compatibility layer: every Phase 3-9 test and script calls `buildServer({ db, network, ... })` completely unchanged and gets back a server whose bare `/api/v1/...` alias behaves exactly as before (plus a harmless, unused `/api/v1/<network>/...` sibling tree). New Phase 10a callers instead pass `networks: { testnet: {...}, mainnet: {...} }` and `defaultNetwork`.
- **The top-level `app` returned by `buildServer` is *also* decorated with the default network's bindings** (`decorateNetwork(app, defaultNetwork, defaultBinding)`, before any child scope exists), purely so `app.db`/`app.network`/etc. keep working for every existing test that reaches into the returned app object directly (`ctx.app.db.prepare(...)` etc., all over `proofs.test.ts`/`webhooks.test.ts`) — these decorators are otherwise unused by request handling, since routes only ever run inside a prefixed or aliased child scope.
- **`/health`/`/stats`'s cross-network `networks` breakdown is opt-in per scope (`decorateNetwork`'s `withCrossNetworkBindings` flag), set only on the alias scope, not on the top-level `app` or the network-prefixed scopes.** Fastify decorators are inherited down the whole prototype chain, so decorating `networkBindings` on `app` itself would have leaked it into every prefixed mount too (confirmed by an initial attempt that did exactly this and broke the "`/api/v1/testnet/stats` reports only testnet" contract) — it must be attached only where it's meant to be read.
- **OpenAPI's `servers` array lists the plain `/api/v1` entry *before* the templated `/api/v1/{network}` one, not after.** `@fastify/swagger` derives every route's documented path by stripping only `servers[0].url` (substituting the template's `default` value first, if `servers[0]` is templated) — putting the template first meant it substituted `default: "testnet"`, computed `/api/v1/testnet` as the string to strip, and *that* silently swallowed the entire testnet-prefixed tree into the same path keys as the alias (confirmed by reproducing it in isolation before fixing the order — see `build.multiNetwork.test.ts`'s `"OpenAPI documents the network path via a templated server variable"` test, which pins the fixed behavior). With the plain entry first, stripping is uniform and all three trees (`/projects` alias, `/testnet/projects`, `/mainnet/projects`) appear as distinct path keys; the templated entry is still present in `servers` to document the `{network}` parameter itself.
- **`server/run.ts` mounts testnet+mainnet together only when `VERICELL_NETWORK` resolves to one of those two; `devnet` keeps the pre-10a single-network path.** Devnet is offckb-local-testing-only (TECHNICAL.md §11) and only ever runs one process at a time (`scripts/e2e.sh`, CI); dual-mounting there would mean constructing real testnet/mainnet `ccc.Client`s (and, if `/health`'s cross-network check ever called them) reaching out to public RPC endpoints from what's supposed to be an offline-friendly local/CI test run, for a network devnet never actually needs. The mainnet startup warning and the `MAINNET_CONFIRM` custodial gate both key off "is the mainnet side active" (always true in dual-network mode, since mainnet is one of the two networks, not just whichever one aliases the root) rather than off `NETWORK === "mainnet"` — a deployment with `VERICELL_NETWORK=testnet` still has a live mainnet DB/client mounted alongside it and must still warn/gate accordingly.
- **`DB_PATH` (a single explicit-file override, pre-10a) is rejected outright in dual-network mode rather than silently reused for both networks.** Passing it through unchanged would point both `openDb(undefined, "testnet")` and `openDb(undefined, "mainnet")` at the identical file, silently merging what's supposed to be two fully isolated databases. `server/run.ts` throws a clear startup error naming `DB_DIR` (which still produces per-network `vericell.<network>.sqlite` filenames) as the fix, rather than picking one network to "win" the shared file.
- **Chain-lookup/custodial-signer caches (`chainLookup.ts`'s `getClient`, `chainClient.ts`'s `getOrMakeClient`/signer cache) became `Map<Network, ...>` keyed by network, not a single module-level value.** Pre-10a, one process only ever ran one network, so a lone cached `ccc.Client`/signer was correct; once one process serves testnet and mainnet at once, that same cache would have handed mainnet's requests a testnet client (or vice versa) depending on which network happened to trigger the lazy build first — each `makeDefault*(network)` call now returns a closure over that network's own cache entry, shared correctly across the several call sites (per-scope default, and the top-level `app`'s mirror decoration) that end up requesting the same network.

## Phase 10b — `packages/web`: launch readiness (network toggle, consumption UX, legal pages)

- **The network is now runtime state in the web app, not a build-time constant — `state.network`, initialized from `core`'s `NETWORK` (baked in via `VITE_VERICELL_NETWORK`) but overridable via the top-bar badge, with the override persisted in `localStorage` under `vericell:network`.** This is a deliberate narrowing of Phase 7's decision ("the network `<select>` is gone... a web build now targets exactly one network"), reversed by this phase's explicit brief. `core.NETWORK`/`isMainnet()` remain the single source of truth for the *default*; nothing outside `network.ts` hardcodes a URL or RPC endpoint — the toggle only ever chooses between the two literal strings `"testnet"`/`"mainnet"`, matching the spec's "toggles testnet ⇄ mainnet" (not devnet, which stays a build-only, non-interactive concern for local offckb testing).
- **`explorerUrlForNetwork(network)` added to `core/network.ts`/exported from the package root**, alongside the existing build-time-only `EXPLORER_URL`. The runtime toggle needs to resolve explorer links for whichever network the visitor is currently on, not just the one baked in at build time; keeping the network→URL map exported from `network.ts` (rather than duplicating it in `web`) preserves the "only `network.ts` may hardcode a network" global rule.
- **API calls are re-based under `/api/v1/{network}/...` on every call (`apiUrlBase()`, reading `state.network` fresh each time), not memoized at connect time.** `packages/api`'s Phase 10a dual-network mounts already serve exactly this path shape per network; the web app previously only ever hit the unprefixed `/api/v1/...` alias (the *default* network's data). A network switch takes effect on the very next fetch with no page reload and no stale closures to invalidate.
- **Switching networks force-disconnects the wallet (`state.signer = null`) rather than trying to detect whether the connected JoyID session is still valid for the new chain.** JoyID/CCC signers are constructed against a specific `ccc.Client`; reusing one across a client swap risks signing against the wrong chain's cell-dep set. Simpler and safer to require an explicit reconnect, surfaced via the `createGate` text ("Switched to MAINNET — reconnect your wallet...") — matches the brief's own "disconnect + clear connection state, prompt to reconnect."
- **The open project-detail view is closed (not re-fetched) on a network switch; search re-queries with the same text.** A detail view is anchored to one specific tx/unid on one specific chain — there is no cross-network mapping to follow, so re-querying it on the new network would either 404 or (worse, if a hash collision-free UNID happened to coincide) show an unrelated project. Search, by contrast, is a free-text/hash query that's meaningful to simply re-run against the new network's index.
- **`anchorProof`/`buildManifest` now thread `genesis`/`prevTxHash` through into the on-chain manifest for a version-mode anchor** (`enterVersionMode` already captured both from the detail view, but nothing previously passed them to the tx that actually got signed) — found while wiring the version-consumption UX banner, which needs to *display* the same version number/genesis it must also *write*. Without this fix every "new version" anchored from the web UI would have produced a manifest missing `genesis`/`prev`, breaking the on-chain version-chain invariant TECHNICAL.md §4 depends on, for every proof anchored through the browser (the CLI's non-custodial `anchor --prev` path was unaffected — separate code path, already correct).
- **Withdraw is implemented directly in `web/main.js` (a small `ccc.Transaction.from({inputs:[...]}); tx.completeFeeBy(state.signer, ...)`), not by importing `chain`'s `buildWithdrawTx`.** Matches Phase 7's existing precedent for `anchorProof` (also hand-built rather than calling `chain.buildAnchorTx`) and its stated reason: `chain/client.ts` imports `node:fs` (for devnet's system-scripts file), which doesn't bundle cleanly for a browser target. `web` still depends only on `core` + `@ckb-ccc/ccc`, not `chain`.
- **"Publish new version" and "Withdraw" are gated by comparing the connected wallet's address to the project's recorded owner address (`isConnectedOwner`), shown as an `alert()` if they don't match — not hidden outright for a non-owner viewer.** The buttons stay visible so anyone can see a project supports versioning/withdrawal at all (informational), but clicking with the wrong wallet connected fails fast with a clear message instead of silently building a transaction that would only fail much later at signing/broadcast.
- **Withdraw's retype-to-confirm compares the input verbatim against the project's title (case- and whitespace-sensitive), not a normalized/trimmed comparison.** The brief asks for the strongest confirmation of the three actions (plain anchor: none; new version: one inline confirm; withdraw: retype-to-confirm) — an exact match is simpler to reason about and harder to fat-finger past than a fuzzy one, consistent with why this action gets the strongest gate in the first place.
- **The version-chain "cell diagram" (`renderCellDiagram`) is a compact visual companion above the existing tx-linked `.version-timeline` list, not a replacement for it.** The existing timeline already carries the information a verifier actually needs to act on (explorer links, block times); the diagram is purely the "at a glance ⊗/●/○ shape of the chain" the brief asks for, reusing the timeline's own per-version status classification (`live`/`consumed`/`pending`) so the two never disagree.
- **Legal pages (`impressum.html`, `datenschutz.html`, `haftungsausschluss.html`) are separate Vite multi-page entries (`vite.config.js`'s `build.rollupOptions.input`), not files dropped in `public/`.** A `public/`-served static HTML file can't reference `/src/style.css` after a production build (that path only exists in dev; the built CSS gets a content-hashed filename under `assets/`) — a proper Vite HTML entry point gets the same processing, hashing and single shared CSS bundle as `index.html`, confirmed by the build output (`impressum.html`/`datenschutz.html`/`haftungsausschluss.html` all reference the identical `assets/style-*.css`).
- **Impressum/Datenschutzerklärung/Haftungsausschluss are placeholder-first, not fabricated.** Every field that legally requires the operator's real identity (name, address, contact, register entry, VAT ID, supervisory authority) is a visibly styled `.fill-in` span reading `FILL_IN: ...`, per the brief — this is content no one but the deploying operator can truthfully supply, and inventing plausible-looking values would be worse than an obvious placeholder. The architecture-description portions (local hashing, no accounts/cookies/tracking, `localStorage` usage, server access logs, API-key hashing, on-chain permanence vs. Art. 17 GDPR erasure) are written to match this codebase's actual behavior as verified while implementing the rest of this phase, not generic boilerplate.
- **The Phase 10 packaging rule ("the strings `testnet`/`mainnet` may appear only in `network.ts`, tests, docs and compose files") is knowingly not upheld in `web/src/main.js` from this phase onward.** That rule predates the runtime-toggle requirement; a client-side testnet⇄mainnet switch cannot exist without comparing against those two literal strings somewhere outside `network.ts` (`state.network === "mainnet" ? "testnet" : "mainnet"` in `wireNetworkSwitch`, `resolveInitialNetwork`'s validation, etc.). `network.ts` remains the only place a network is mapped to a URL/RPC endpoint or otherwise "hardcoded" in the sense the original rule cared about; Phase 10a's own dual-network route-prefix strings already established this same precedent server-side.
